# План логирования подключения Android-клиента (кнопка «Доступность всех серверов»)

## 1. Контекст

Сейчас при нажатии кнопки «Доступность всех серверов» пользователь видит только итог ошибки (`HTTP 401`) без детализации этапа сбоя.
Для работы в ветке `develop` нужен единый документ, чтобы добавить трассировку процесса подключения и ускорить диагностику проблем с Base URL, токеном и сетью.

## 2. Цель

Организовать сквозное логирование запроса Android-клиента:

1. старт действия в UI;
2. подготовка запроса;
3. отправка HTTP-запроса;
4. получение ответа;
5. обработка результата в UI.

Результат: по логам можно быстро понять, где именно падает цепочка и почему пользователь получает 401/timeout/SSL ошибки.

## 3. Что логируем (минимальный состав)

## 3.1 Android-клиент

События (уровень `INFO`/`WARN`/`ERROR`):

- `availability_all_click_started`
- `availability_all_request_prepared`
- `availability_all_request_sent`
- `availability_all_response_received`
- `availability_all_result_rendered`
- `availability_all_failed`

Поля:

- `request_id` (UUID на один клик)
- `timestamp_utc`
- `base_url_host` (без path/query)
- `endpoint` (например `/api/v1/availability/all`)
- `http_method`
- `http_status` (если есть)
- `duration_ms`
- `network_type` (wifi/mobile/unknown)
- `token_present` (`true/false`, без значения токена)
- `error_type` (`HTTP_401`, `TIMEOUT`, `SSL`, `NO_NETWORK`, `UNKNOWN`)
- `error_message_short` (без персональных данных)

Требования безопасности:

- не логировать токен, полный URL c query и персональные данные;
- не логировать тело ответа целиком;
- для ошибок ограничивать длину сообщения (например 300 символов).

## 3.2 Backend/BFF

Для корреляции добавить/пробросить `X-Request-ID` и писать в серверные логи:

- входящий `request_id`;
- код ответа;
- длительность обработки;
- причина отказа авторизации (классифицированно, без утечки чувствительных данных).

## 4. Техническая реализация в `develop`

## Шаг 1. Корреляционный идентификатор

- На Android генерировать `request_id` при каждом нажатии кнопки.
- Добавлять `X-Request-ID` в заголовок запроса.

## Шаг 2. Сетевой интерсептор

- Добавить OkHttp Interceptor для логирования «до/после» запроса.
- Для релиза оставить структурированные безопасные логи, без debug-шума.

## Шаг 3. Логирование UI-этапов

- В обработчике кнопки и ViewModel логировать этапы выполнения и финальный статус отображения.

## Шаг 4. Нормализация ошибок

- Маппинг исключений/кодов в единый справочник `error_type`.
- Для `401` отдельная запись с рекомендацией проверить Base URL и токен.

## Шаг 5. Серверная корреляция

- На BFF принимать `X-Request-ID` и писать его в access/app logs.
- Если заголовок отсутствует — генерировать серверный id и возвращать в ответе.

## 5. Приёмочные критерии

1. Для одного нажатия кнопки в логах Android есть минимум 4 записи: `started`, `request_sent`, `response_received|failed`, `result_rendered`.
2. Во всех записях одного запроса одинаковый `request_id`.
3. При `401` в логе есть `error_type=HTTP_401`, `http_status=401`, без токена.
4. На backend можно найти запись по тому же `request_id`.
5. Логи не содержат секретов (Bearer token, персональные данные, полный response body).
